# BuildFlow v1.2 Dashboard Task Plan

## Overview

Canonical task plan for the BuildFlow v1.2 dashboard implementation. This document defines the ordered implementation sequence, file scopes, and acceptance criteria for each task.

The goal is to move the dashboard from a basic functional state to a premium, calm, and visually cohesive experience that communicates BuildFlow's core promise: "BuildFlow has your sources, knows your project context, and is ready to help you continue the build plan."

**Do not skip tasks.** Each task prepares the foundation for the next. Tasks 1–2 establish structure; tasks 3–7 add functionality and visual polish; tasks 8–11 finalize and verify.

## Service Health and Verification Protocol

**Local Web Dashboard (port 3054):**

Every dashboard implementation task should end with a healthy web service. Use this protocol:

1. **Before cleanup:** If the dev server is already running and healthy, do not restart unnecessarily. Check:
   ```bash
   curl -sS http://127.0.0.1:3054/api/openapi | head -20
   # Should return valid JSON, not module/chunk errors
   ```

2. **If unhealthy or stale:**
   - Use targeted port cleanup (never broad `pkill`):
     ```bash
     for port in 3054; do
       pids=$(lsof -tiTCP:$port -sTCP:LISTEN || true)
       if [ -n "$pids" ]; then kill $pids || true; fi
     done
     ```
   - For dashboard code changes, use `pnpm local:rebuild-web` when the local web server may already be running
   - Never run `pnpm --dir apps/web build` while `pnpm --dir apps/web dev` is still running on port 3054
   - Only remove `.next/` if there are module/chunk errors or the build is clearly stale
   - Stop web first, clear `.next/`, then build with web stopped, then restart web
   - Rebuild manually only after stopping web on 3054 first
   - Restart: `pnpm --dir apps/web dev > /tmp/buildflow-web.log 2>&1 &`
   - Wait 3–4 seconds for server to start
   - Verify:
     ```bash
     curl -sS http://127.0.0.1:3054/api/openapi | head -20
     ```

3. **Expected result:**
   - OpenAPI JSON returned (schema version 3.1.0, title "BuildFlow API")
   - No "Cannot find module" or chunk errors
   - Port 3054 is responsive

4. **If verification fails:**
   - Check logs: `tail -20 /tmp/buildflow-web.log`
   - Verify no other process is bound to 3054
   - Ensure `.next/` build is fresh
   - Run `pnpm install` if dependencies are stale

**Safe verification sequence for BuildFlow local and public surfaces:**

1. `curl -sS http://127.0.0.1:3052/health | head -20`
2. `curl -sS http://127.0.0.1:3053/health | head -20`
3. `curl -sS http://127.0.0.1:3054/api/openapi | head -20`
4. `curl -sS http://127.0.0.1:3054/api/actions/status | head -20`
5. `curl -i -sS https://buildflow.prochat.tools/api/openapi | head -80`
6. `curl -i -sS https://buildflow.prochat.tools/api/actions/status | head -80`
7. For repeated verification, use `pnpm local:verify`

**Type-check and Build Verification:**

- Run `pnpm --dir apps/web type-check` after any dashboard code change
- Run `pnpm --dir apps/web build` after any dashboard code change
- Both must pass with zero errors
- Do not skip even if the change "looks small"
